# 如何把千里马计划合并到大模型

## 一句话说明

千里马计划不是把文件直接塞进大模型参数里，也不是训练一个新模型。正确做法是把千里马计划作为大模型外部的工作台、规则层、上下文包和工具层接入。

大模型负责理解、推理和生成；千里马计划负责管理文件、数据源、workflow、权限、成本、上下文压缩和执行记录。

## 合并方式

```text
大模型
  -> 运行启动索引脚本
  -> 读取千里马工作区索引
  -> 按规则选择任务卡和 workflow
  -> 按上下文策略读取或压缩文件
  -> 调用数据源和工具
  -> 生成报告或行动建议
  -> 写入日志、成本和反馈
  -> 记录 input/output tokens 和估算费用
  -> 进入 Loop Engineering 改进下一轮
```

也可以理解成：

```text
模型 = 大脑
千里马计划 = 工作台 + 记忆目录 + 操作规程 + 安全边界
```

## 不要这样做

不要把整个 `.qianlima/` 一次性塞进 prompt。

原因：

- 文件会越来越多，容易超出上下文。
- 模型会被无关资料干扰。
- 成本会升高。
- 重要规则可能被长文本稀释。
- 大模型不应该长期记住敏感数据。

正确做法是按需加载：

```text
先运行 start-qianlima.ps1
再读 .qianlima/WORKSPACE_INDEX.md
再读 work.ws
再按用户任务读 task-card / workflow / data-source
长文件先压缩
高风险动作前回读原始来源
```

## 必读文件顺序

每次启动千里马计划时，大模型必须先生成索引，再读少量入口文件。

### 第 0 层：强制索引

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File ".\start-qianlima.ps1"
```

生成后先读：

```text
.qianlima/WORKSPACE_INDEX.md
```

如果索引不存在、缺文件或明显过期，不开始任务，先重新运行启动脚本。

### 第 1 层：工作入口

```text
.qianlima/README.md
.qianlima/work.ws
.qianlima/workflow-index.yaml
```

作用：

- 知道这是哪个工作区
- 用户当前关注什么
- 有哪些可用 workflow

### 第 2 层：安全和上下文

```text
.qianlima/risk-rules.yaml
.qianlima/context-policy.yaml
.qianlima/model-adapters.yaml
```

作用：

- 知道哪些动作不能自动执行
- 知道文件该怎么读、怎么压缩
- 知道当前模型该使用什么上下文预算

### 第 3 层：按任务加载

根据用户需求再读取对应文件。

例如用户说“我要做竞品对比”：

```text
.qianlima/task-cards/competitor-comparison.yaml
.qianlima/templates/competitor-comparison_template.md
.qianlima/data-sources.yaml
.qianlima/file-registry.yaml
```

例如用户说“先帮我整理这批资料”：

```text
.qianlima/task-cards/knowledge-digest.yaml
.qianlima/workflows/knowledge_digest.yaml
.qianlima/context-policy.yaml
.qianlima/templates/knowledge-digest_template.md
```

## 系统提示词模板

可以把下面这段作为大模型的系统提示词或项目说明。

```text
你正在运行“千里马计划”工作区。

你的目标不是自由聊天，而是按千里马工作区的规则帮助用户完成工作任务。

启动时先读取：
0. 先运行 start-qianlima.ps1 生成 .qianlima/WORKSPACE_INDEX.md
1. .qianlima/WORKSPACE_INDEX.md
2. .qianlima/README.md
3. .qianlima/work.ws
4. .qianlima/workflow-index.yaml
5. .qianlima/risk-rules.yaml
6. .qianlima/context-policy.yaml
7. .qianlima/model-adapters.yaml

当用户提出任务时：
1. 先判断是否匹配 task-cards。
2. 再读取对应 workflow 和 template。
3. 如果需要数据，读取 data-sources.yaml 和 file-registry.yaml。
4. 如果文件很多或文件很长，必须按 context-policy.yaml 自动压缩。
5. 高风险动作只给建议，不能自动执行。
6. 生成结果时必须说明数据来源、待验证项和使用成本。
7. 执行后把日志、成本和反馈写入对应目录。
8. 每次任务结束后，把 input tokens、output tokens、模型名、估算费用写入 .qianlima/usage-ledger/。
9. 每次任务都进入 Loop Engineering：执行、评估、分析、改进、记录、再执行。

默认面向没有技术背景的大众用户，用业务语言解释，不要求用户理解目录、脚本、API 或模型上下文。
```

## Token 与费用记录

千里马计划把 token 和费用当成工作治理的一部分。每次任务结束后，都要记录：

```text
模型供应商
模型名
input tokens
output tokens
cached input tokens
reasoning tokens（如果模型提供）
估算费用
是否使用上下文压缩
读取了哪些文件
生成了哪个结果文件
```

记录目录：

```text
.qianlima/usage-ledger/
```

记录模板：

```text
.qianlima/templates/token-usage-record_template.yaml
```

当前公开模板的最小启动包估算为 **10,655 到 19,534 input tokens**。如果工作区文件变多，先重新运行启动脚本，再重新估算。

费用按供应商当前价格计算，不在千里马计划里写死：

```text
估算费用 =
输入 tokens / 1,000,000 × 输入单价
+ 输出 tokens / 1,000,000 × 输出单价
+ 缓存输入 tokens / 1,000,000 × 缓存输入单价
```

如果模型或平台不返回精确 token，用保守估算并标注 `estimated: true`。

## Loop Engineering 接入方式

Loop Engineering 是千里马计划的持续改进机制。

```text
执行 -> 评估 -> 分析 -> 改进 -> 记录 -> 再执行
```

接入大模型时，不要把一次任务当成孤立问答。任务完成后必须检查：

- 输出是否满足模板
- 数据源是否清楚
- 风险是否说明
- token 花费是否异常
- 用户是否修改或拒绝结果
- 是否需要更新规则、模板、字段映射或用户偏好

正式说明见：

```text
Loop Engineering 循环工程.md
.qianlima/improvement-loop.yaml
.qianlima/evaluation-tasks.yaml
```

文档分组和阅读顺序见 `docs/README.md`。

## DeepSeek 接入方式

DeepSeek 是千里马计划优先适配的大模型。

### 推荐模型选择

| 场景 | 推荐 |
|---|---|
| 日报、批量报告、资料消化 | `deepseek-v4-flash` |
| 复杂推理、多文件综合、高价值决策 | `deepseek-v4-pro` |
| 兼容旧调用 | `deepseek-chat` / `deepseek-reasoner` 作为别名过渡 |

具体策略见：

```text
.qianlima/model-adapters.yaml
```

### DeepSeek 使用原则

- 即使 DeepSeek 支持超长上下文，也不要把全部文件一次性塞进去。
- 先用 task-card 和 workflow 缩小范围。
- 长文件进入 `context-summaries/`。
- 稳定系统规则和稳定摘要放在前缀，方便缓存。
- thinking 模式保留更大安全冗余。
- 不要把 reasoning content 重新塞回下一轮上下文。

### DeepSeek 推荐启动包

```text
.qianlima/README.md
.qianlima/work.ws
.qianlima/workflow-index.yaml
.qianlima/risk-rules.yaml
.qianlima/context-policy.yaml
.qianlima/model-adapters.yaml
```

这 6 个文件是 DeepSeek 运行千里马计划的最小启动包。

## OpenAI / Anthropic / Google 接入方式

这几类模型不在千里马计划中硬编码具体上下文窗口。

原因：

- 模型版本变化快。
- 同一厂商不同模型窗口不同。
- 应从运行环境或 API 配置读取真实限制。

接入时只需要做两件事：

1. 在运行环境声明模型名、上下文窗口、最大输出。
2. 让 `.qianlima/model-adapters.yaml` 选择对应 capability class。

如果无法确认模型能力，使用保守默认策略：

```text
usable_context_ratio: 0.60
safety_reserve_ratio: 0.40
summarize_large_files: true
```

## 本地模型和开源模型接入方式

本地模型差异很大，不能默认认为它支持长上下文。

必须手动声明：

```yaml
provider: local_or_open_weight
model: your_model_name
total_window_tokens: 32768
max_output_tokens: 4096
supports_tools: false
supports_json: true
```

如果没有工具调用能力，就只让它做：

- 摘要
- 分类
- 报告草稿
- 问题提取

不要让它直接执行：

- 写文件
- 调 API
- 修改广告
- 改预算
- 外发消息

## 普通用户怎么用

普通用户不需要知道模型怎么接。

用户只需要说：

```text
我要做竞品对比。
帮我优化这个 Listing。
算一下这个产品赚不赚钱。
跑一下这些关键词排名。
帮我判断这个品类能不能做。
先帮我整理这批资料。
```

大模型应该自动：

1. 匹配任务卡。
2. 读取对应 workflow。
3. 检查数据源。
4. 压缩必要上下文。
5. 生成报告。
6. 标注来源、风险和成本。

## 项目接入步骤

### 第 1 步：把 `.qianlima/` 放进工作目录

推荐结构：

```text
你的工作目录/
  .qianlima/
  原始文件/
  报告/
```

### 第 2 步：让模型先读启动包

启动包：

```text
.qianlima/README.md
.qianlima/work.ws
.qianlima/workflow-index.yaml
.qianlima/risk-rules.yaml
.qianlima/context-policy.yaml
.qianlima/model-adapters.yaml
```

### 第 3 步：用户说任务

例如：

```text
我要做竞品对比：B08MPY943X、B0CLRPJVG7、B0DK1SLSYL，站点 US。
```

### 第 4 步：模型按任务卡执行

读取：

```text
.qianlima/task-cards/competitor-comparison.yaml
.qianlima/templates/competitor-comparison_template.md
.qianlima/data-sources.yaml
.qianlima/file-registry.yaml
```

### 第 5 步：输出并记录

输出到：

```text
.qianlima/reports/
.qianlima/logs/
.qianlima/usage-ledger/
.qianlima/feedback/
```

## 最小运行清单

大模型要真正“合并”千里马计划，至少需要做到：

- 能读取 `.qianlima/README.md`
- 能读取 `work.ws`
- 能根据 `workflow-index.yaml` 匹配任务
- 能遵守 `risk-rules.yaml`
- 能按 `context-policy.yaml` 压缩上下文
- 能按 `model-adapters.yaml` 选择模型预算
- 能读取 task-card 和 template
- 能写入 reports、logs、usage-ledger

## 验收标准

1. 用户说自然语言任务时，模型能匹配正确 task-card。
2. 模型不会要求用户理解内部目录。
3. 模型不会一次性读取所有文件。
4. 模型会说明数据来源和待验证项。
5. 模型会保留上下文安全冗余。
6. 模型不会自动执行高风险动作。
7. 模型会记录使用量和成本。

## 当前建议

第一版优先接 DeepSeek：

```text
deepseek-v4-flash：普通日报、批量报告、资料消化
deepseek-v4-pro：复杂推理、多文件综合、高价值决策
```

跑通后再把同一套 `.qianlima/` 迁移到 OpenAI、Anthropic、Google 或本地模型。
