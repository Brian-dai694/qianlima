# User Profile and Work Scenario Discovery 用户画像与工作场景发现

## 结论

面对普通用户，千里马不能先要求他管理文件，再让系统理解他在做什么。正确顺序是：

1. 先接收文档、知识库、聊天记录或会议材料
2. 先做资料消化，提炼可读摘要
3. 再从资料和对话中抽取用户画像
4. 再识别工作场景
5. 最后把结果写回工作区，让后续 workflow 直接使用

这个流程的目标不是给用户增加操作，而是让系统自己把“我是谁、我在做什么、我常做什么”整理出来。

## 为什么需要这一步

普通用户通常不会主动维护文件结构，也不会想要先建目录、再建索引、再建字段。

他们的真实表达通常是：

- 帮我看这些资料
- 先整理一下我最近都在忙什么
- 你看看我平时主要做哪些事
- 以后我这种工作也可以直接交给你吗

所以系统不能把“文件治理意识”当成前提，而要把它当成后置结果。

## 推荐链路

```text
用户提供资料 / 文档 / 知识库 / 对话记录
  -> NotebookLM 或资料消化 workflow 先做摘要
  -> 画像提取器识别角色、目标、常用工具、输出偏好
  -> 场景发现器提炼重复工作、固定任务和高频数据源
  -> 系统给用户确认
  -> 写入 user-profile.yaml / work.ws / 相关 summary
```

## 画像和场景的区别

画像回答的是“这个人是谁”。

场景回答的是“这个人反复在做什么”。

画像更稳定，场景更动态。

示例：

- 画像：Amazon 运营负责人，偏好表格和 Markdown，外部写入要确认
- 场景：每日广告运营日报、关键词监控、库存预警、利润复盘

## 输入来源

画像和场景判断可以来自这些材料：

- 聊天对话
- 上传文档
- 会议纪要
- NotebookLM 摘要
- 知识库页面
- 任务卡历史
- 数据源和报告内容

优先级不是谁更“技术”，而是谁更能说明用户真实工作。

## 普通用户问题模板

系统不应该问一堆技术问题，而应该用业务语言问最少的问题。

建议只问这些：

1. 你平时主要做哪类工作
2. 哪些事经常重复
3. 你希望最后得到什么输出
4. 哪些动作不能自动执行
5. 哪些资料以后要固定参考

如果资料已经足够，很多问题可以直接从文档和历史对话里推断，不必重复问。

## 用户画像模板

建议写入 `.qianlima/user-profile.yaml`。

```yaml
user_profile:
  user_id: current_user
  inferred_role: Amazon 运营负责人
  confidence: medium
  work_style:
    primary_channels:
      - 文档
      - 飞书
      - 表格
    preferred_outputs:
      - Markdown 报告
      - 表格
      - 行动项清单
    review_style:
      - 先摘要
      - 再结论
      - 最后风险
  constraints:
    no_file_management_skill: true
    prefers_plain_language: true
    avoid_manual_configuration: true
    confirm_before_external_write: true
  discovered_scenarios:
    - id: ad_ops
      confidence: high
    - id: keyword_tracking
      confidence: medium
```

## 工作场景模板

建议写入 `work.ws` 或对应的场景总结文件。

```yaml
scenario:
  id: ad_ops
  name: 广告运营
  frequency: daily
  trigger_signals:
    - 每天看广告消耗
    - 经常提 ACoS / CPC / CVR
    - 反复要求整理日报
  required_inputs:
    - 广告数据
    - 销量数据
    - 关键词表现
  preferred_outputs:
    - 日报
    - 异常提醒
    - 调整建议
  confidence: high
```

## 场景发现规则

系统在下面情况应该主动建议登记场景：

- 同一类任务 7 天内出现 3 次以上
- 用户说“每天”“每次”“以后都”
- 用户重复要求同一种输出
- 文档里已经出现固定数据源和固定报告格式
- 对话里已经能看出固定角色和固定指标

但系统不能直接替用户下最终定义，必须先确认。

## 确认方式

系统应该输出短结论，而不是长解释。

例如：

```text
我判断你更像是一个跨境电商运营，常见工作是广告、关键词、库存和利润。
我还看到你有一类固定任务：每天看广告消耗并生成日报。
要不要把这类任务登记成固定工作场景？
```

## 和 NotebookLM 的关系

NotebookLM 负责先把材料变成可读摘要。

千里马负责在摘要基础上进一步做三件事：

- 抽用户画像
- 找工作场景
- 进入 workflow 和治理体系

所以 NotebookLM 不是终点，它是画像和场景发现的前置入口。

## 和知识库的关系

知识库不是让用户自己维护文件树，而是给系统提供长期可复用的上下文。

对于普通用户，知识库最重要的不是“文档分类”，而是能不能帮助系统知道：

- 他是谁
- 他常做什么
- 他最近在忙什么
- 哪些内容应该长期记住

## 存储建议

建议分三层保存：

1. `context-summaries/` 存提取过程的摘要
2. `.qianlima/user-profile.yaml` 存用户画像
3. `work.ws` 和 `work-hub.ws` 存工作场景和联动关系

## 成功标准

这件事做对了，应该满足：

- 用户不需要先学会文件管理
- 系统能从资料里推断角色和场景
- 系统能给出少量明确确认问题
- 系统能把画像和场景写进工作区
- 后续任务能直接复用这些结果

一句话：

**先从资料里认识用户，再把用户的真实工作场景固定下来。**
