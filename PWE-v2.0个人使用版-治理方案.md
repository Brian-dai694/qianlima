# PWE v2.0 个人使用版治理方案

> **定位**：个人多项目（代码仓库）治理控制台  
> **目标**：让一个人用多个 Agent 管好多个代码项目，减少重复检查、重复沟通和重复踩坑  
> **核心原则**：保留有用的治理能力，降低平台化复杂度

---

## 一、核心定位

个人版 PWE 不追求做一个完整的团队级平台，而是作为个人开发者的项目治理中枢。

它要解决的问题是：

- 我现在有哪些项目？
- 每个项目的真实状态是什么？
- 哪些地方违反了规则？
- 哪些任务可以交给 Agent 做？
- 多个项目之间有没有接口、规范、经验可以同步？

一句话概括：

**PWE 个人版 = project.ws + audit + loop + skills + 多 Agent + 轻量跨项目通信。**

---

## 二、个人版能力结构

个人版保留 5 个核心层次：

```text
第一层：项目状态
project.ws / 项目索引 / 项目初始化

第二层：治理检查
约束规则 / audit / 自动修复建议

第三层：执行循环
LoopEngine / 测试 / Review / Commit

第四层：多 Agent 协作
主控 Agent / 执行 Agent / 审计 Agent / 文档 Agent

第五层：跨项目通信
接口变更提醒 / 经验复用 / 统一规范 / 联动任务
```

相比平台版，个人版暂时不做复杂 UI、WebSocket、A2A 协议和重型 Connector。

---

## 三、项目状态

每个项目保留一个 `.ws/project.ws`，作为该项目的唯一真相源。

它记录：

- 项目实体：模块、测试、文档、脚本、Agent、Skill
- 项目关系：依赖、调用、覆盖、归属
- 约束状态：哪些规则通过，哪些规则失败
- 执行记录：最近的 audit、loop、修复记录
- 项目摘要：当前目标、风险、下一步

个人使用时，不需要一开始就把所有内容建得很完整。优先记录三类信息：

- 重要代码模块
- 关键测试和文档
- 当前正在推进的任务

---

## 四、治理检查

个人版保留约束系统，但规则要少而实用。

建议默认规则：

| 规则 | 说明 |
|---|---|
| C001 | project.ws 必须存在并能被读取 |
| C002 | 重要脚本、工具、入口必须登记 |
| C003 | 新功能必须有对应测试或明确豁免 |
| C004 | 重要模块必须有说明文档 |
| C005 | 修改后必须能跑通基础检查 |
| C006 | project.ws 与实际文件结构不能明显不一致 |

个人版不需要追求 26 条规则一次性全部启用。先用少量高价值规则跑顺，再逐步增加。

---

## 五、执行循环

LoopEngine 在个人版里主要做三件事：

1. 执行任务前读取项目状态
2. 执行任务后跑测试和审计
3. 根据结果决定是否继续修复

推荐保留三种模式：

| 模式 | 用途 |
|---|---|
| single | 日常小任务，跑一轮后停下 |
| continuous | 修 bug、清理违规，直到收敛 |
| incremental | 只检查最近改过的文件 |

个人版默认使用 `single`，遇到连续修复类任务再用 `continuous`。

---

## 六、多 Agent 协作

个人版需要多 Agent，但不需要复杂协议。

推荐配置：

| Agent | 职责 |
|---|---|
| 主控 Agent | 拆任务、分配工作、决定是否继续 |
| 执行 Agent | 改代码、加功能、修 bug |
| 审计 Agent | 查约束、跑测试、找遗漏 |
| 文档 Agent | 更新 README、设计文档、变更记录 |

工作流示例：

```text
用户提出需求
  ↓
主控 Agent 读取 project.ws
  ↓
执行 Agent 修改代码
  ↓
审计 Agent 跑检查
  ↓
文档 Agent 更新说明
  ↓
主控 Agent 汇总结果
```

个人版的关键不是让 Agent 互相聊天，而是让它们围绕同一个项目状态协作。

---

## 七、跨项目通信

跨项目通信要保留，但要做轻量版。

它的作用是：当你维护多个相关项目时，一个项目的变化可以被另一个项目知道。

适合个人使用的四类场景：

| 场景 | 例子 |
|---|---|
| 接口变更提醒 | 后端 API 字段变了，前端项目收到提醒 |
| 经验复用 | 项目 A 修过的错误，项目 B 遇到时能查到解决办法 |
| 统一规范 | 多个项目共用提交规范、目录规范、测试要求 |
| 联动任务 | 后端完成支付接口后，前端自动生成接入任务 |

个人版不需要完整 A2A 协议。建议用一个轻量的 `hub.ws` 做跨项目索引。

目录结构：

```text
.pwe/
  hub.ws
  projects.yaml
  shared-memory/
  shared-rules/
```

`hub.ws` 负责记录：

- 当前接入了哪些项目
- 项目之间是否有关联
- 哪些接口、规范、经验可以共享
- 最近有哪些跨项目事件

跨项目事件示例：

```yaml
event:
  type: api_changed
  source_project: backend
  target_project: frontend
  summary: "User list API added field avatar_url"
  suggested_task: "Update frontend user table and type definition"
```

这样既保留跨项目价值，又不用引入复杂消息总线。

## 八、Skill 系统

个人版 Skill 只保留常用场景。

建议默认 5 个：

| Skill | 用途 |
|---|---|
| pwe-dev-flow | 日常开发 |
| pwe-audit | 项目检查 |
| pwe-debug | 调试排错 |
| pwe-doc | 文档更新 |
| pwe-cross-project | 跨项目同步 |

`AGENTS.md` 里只需要放 Skill 索引，不需要写太长。

示例：

```md
## Skills

- pwe-dev-flow：修改代码、添加功能、修 bug 时使用
- pwe-audit：检查 project.ws、测试、约束状态时使用
- pwe-debug：定位异常、失败测试、状态不一致时使用
- pwe-doc：更新 README、设计文档、变更记录时使用
- pwe-cross-project：处理接口变更、跨项目经验和联动任务时使用
```

## 九、保留与删除

个人版保留：

- `project.ws`
- 基础约束系统
- audit 检查
- LoopEngine
- Skill 索引
- 多 Agent 分工
- 轻量跨项目通信
- 共享记忆

个人版暂缓：

- 重型 UI
- WebSocket 实时同步
- 完整 A2A 协议
- 企业级 Connector
- 复杂权限系统
- 大规模团队协作模型

---

## 十、推荐路线

第一阶段：单项目跑通

- 初始化 `.ws/project.ws`
- 建立基础约束
- 跑通 audit
- 跑通 single loop

第二阶段：多 Agent 分工

- 增加主控 Agent
- 增加执行 Agent
- 增加审计 Agent
- 固化日常开发流程

第三阶段：跨项目通信

- 建立 `.pwe/hub.ws`
- 登记多个项目
- 记录共享规范
- 支持接口变更提醒
- 支持经验复用

第四阶段：再考虑 UI 和自动化

- 有稳定流程后再做可视化
- 有明确需求后再接 GitHub/Jira/飞书等外部系统

---

## 十一、最终形态

个人版 PWE 的最终形态不是一个庞大的平台，而是一个稳定的个人项目控制台。

它应该做到：

- 看得清每个项目的状态
- 管得住代码、文档、测试和规则
- 调得动多个 Agent 协作
- 记得住历史问题和解决办法
- 能在多个项目之间同步关键变化

最终目标：

**一个人，也能用多 Agent 管多个项目，并且不被复杂基础设施拖累。**
